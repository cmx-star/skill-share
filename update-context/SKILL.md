# 更新上下文

根据本轮任务的真实证据维护项目 AI 上下文。不要把固定模板提前复制到所有项目；只有出现对应知识时，才创建目录和文件。

## 触发

在以下情况执行：

- 一轮较复杂的开发已经完成，并且有 diff 与验证结果。
- 对话中确认了长期有效但仓库尚未记录的项目规则、事实或决策。
- 用户明确要求初始化、更新或检查项目 AI 上下文。
- 用户要求总结近期工作、检查重复流程、发现可复用的 Skill、子代理或自动化，或优化现有资产。

以下情况直接输出 `no-op`：

- 只有解释、闲聊、状态查询或一次性操作。
- 只有局部实现变化，没有稳定规则或流程变化。
- 所有候选均为推测，无法由仓库或验证结果确认。

## 证据输入

按以下优先级读取最小必要证据，始终遵循渐进披露并只保留可追溯的结论：

1. 最近的 Codex 会话、任务摘要，以及当前对话中的目标、纠正、决定、验收和未解决风险。
2. 可用的 Codex 记忆与 rollout 摘要，用于发现跨会话模式；没有读取能力时明确说明，不得臆造内容。
3. Chronicle 已启用且可访问时，只将其作为 Codex 外部重复工作的线索。关键事实必须回到原始系统或可验证记录确认；不可访问时跳过。
4. 现有 Skills、自定义子代理和自动化配置，优先复用或最小扩展，避免重建重叠资产。
5. Git 状态、实际 diff、修改文件、适用的 `AGENTS.md`、`AGENTS.local.md`、现有 AI 文档，以及代码、配置、类型、测试和已执行验证结果。

对话只能提供候选线索。项目事实必须由代码、配置、测试、明确团队决定或用户确认支持。

不得输出敏感会话细节、密钥、令牌或完整私有配置。无法访问的证据源只报告为未检查，不得作为否定证据。

## 知识分类

- `AGENTS.md`：项目入口、权威命令、高频边界和按需读取上下文的说明。
- `Rule`：长期有效、可判断遵守或违反、忽略会产生实际风险的规范。
- `Reference`：架构、模块职责、领域术语、模型或状态流转等当前事实。
- `Decision`：存在真实替代方案，并具有跨模块、长期或高逆转成本的选择。
- `Workflow asset candidate`：重复流程已经形成稳定输入、输出、步骤和验证路径，且适合 Skill、子代理或自动化之一。
- `Task-only`：仅对当前任务有效，不持久化。

“约束”不是单独的存储类型：稳定项目约束归为 Rule；任务约束留在当前任务上下文；高频核心 Rule 可以摘要进入 `AGENTS.md`。

## 按需创建项目文件

只创建本轮确实需要的路径，不预建完整目录树：

```text
<project>/
├── AGENTS.md                         # 仅在需要项目入口或按需读取说明时创建
├── AGENTS.local.md                   # 仅出现本机信息时创建，并加入 Git 忽略
├── docs/ai/rules/<topic>.md          # 仅存在 Rule 候选时创建
├── docs/ai/references/<topic>.md     # 仅存在 Reference 更新时创建
├── docs/ai/decisions/<nnn>-<topic>.md# 仅存在重要决策时创建
└── .codex/skills/<name>/             # 仅在用户批准 Skill 晋升后创建
```

规则：

- 不创建空目录、空索引或与项目无关的示例文件。
- 不因为目录规范中出现 `api`、`database` 或 `payment` 就创建这些文件。
- 优先更新已有权威来源；同一事实不得复制到多份文件。
- 模块范围的规则优先写入文档元数据或根 `AGENTS.md`；只有模块确实有独立规则时，才创建目录级 `AGENTS.md`。
- 项目不得依赖某位成员私有的全局 Skill 或配置才能正确执行。

## 创建规则

### AGENTS.md

仅在至少存在一项已验证内容时创建或更新：

- 权威安装、测试、构建或运行命令。
- 全项目高频且稳定的边界。
- 指向 Rule、Reference、Decision 或项目 Skill 的按需读取入口。

不要把未验证技术栈、目录职责或命令写成事实。已有 `AGENTS.md` 时做最小追加，不重写无关内容。

### Rules

Rule 候选必须包含：适用范围、规范文本、证据、验证方式和违反风险。新 Rule 默认标记 `draft` 或作为候选补丁展示；用户或团队确认前不得宣称已经生效。

### References

Reference 必须描述当前已验证事实，并附代码、配置、测试或既有文档路径。可以直接修正明确过期的事实，但必须保留依据并避免复制源码细节。

### Decisions

Decision 必须包含背景、真实候选方案、选择、权衡和后果。只有对话中已经明确作出决定时才能标记 `accepted`；否则使用 `proposed`。

### Workflow Assets

检查编码、调研、写作、规划、沟通、运营、分析和个人事务中重复、耗时、易出错或强依赖上下文的流程。只保留同时满足以下条件的候选：

1. 已至少发生两次，或具有高重复概率且重复成本显著。
2. 输入、可复现步骤、输出或终止条件稳定。
3. 标准化能实质提高速度、质量、一致性或可靠性。
4. 没有已有资产充分覆盖；存在相近资产时优先建议扩展。

选择最小形态：

- `Skill`：可复用的工作流或操作手册。
- `Sub-agent`：可委托、边界清晰的专业角色或调查任务。
- `Automation`：周期检查、报告、提醒或持续监控。
- `Extend existing asset`：既有资产只需补齐窄范围能力。
- `Skip`：一次性、模糊、敏感、证据不足或不适合标准化的工作。

每个保留候选须记录名称、来源和日期、发生频率或置信度、推荐形态、核心理由、触发条件、输入输出、允许写入范围、验证方式和风险。

只有候选证据充分、确实存在缺口，且用户已明确批准对应写入时，才创建或扩展资产。创建项目 Skill 前调用 `skill-creator`；创建子代理或自动化前使用当前环境提供的对应机制。不要把 Chronicle 线索直接变成资产。

## 工作流

1. 建立本轮目标、变更、验证和关键纠正的简短事实清单。
2. 按证据优先级识别重复模式和既有资产覆盖，核验关键来源。
3. 输出第一阶段的简表：候选工作流、证据与日期、频率或置信度、推荐形态和核心理由。明确列出跳过项。
4. 对通过全部门槛的候选，检查是否具有明确的创建或扩展授权；无授权时仅作为候选报告。
5. 只对证据充分且已获授权的缺口执行第二阶段：确定唯一目标、最小改动和验证方式，再创建或扩展资产。
6. 同时维护本轮已经核验的项目上下文：判断事实是否已记录、过期或冲突，并按分类更新唯一的权威来源。
7. 检查变更范围、链接、重复内容、敏感信息和既有资产重叠。
8. 报告实际创建或更新的文件、证据、待审核内容与 `no-op` 分类。

## 输出格式

输出阶段一清单后，以以下固定总结收尾；空分组使用“无”：

- `本次新建或扩展的全部资产`
- `本次主动跳过未打包的工作项`
- `还需更多支撑证据才能完成标准化打包的工作项`

同时按需输出以下执行分组：

- `created`：本轮按需创建的目录和文件。
- `updated`：更新的现有权威来源。
- `candidates`：尚需用户或团队确认的 Rule、Decision 或 Workflow asset。
- `skipped`：没有长期记录的内容及原因，例如临时、重复、可直接从代码读取或证据不足。
- `verification`：结构检查、链接检查、相关项目验证结果及未验证项。

若没有必要更新，明确输出：`no-op：本轮没有需要长期记录的稳定项目知识。`

## 安全边界

- 不写入密钥、令牌、`.env` 内容或完整敏感配置。
- 不从对话推测 API、命令、路径、架构或业务规则。
- 不自动修改个人或组织级规则。
- 不自动把候选 Rule、Decision 或 Skill 宣布为正式团队规则。
- 不覆盖无关用户修改，不执行破坏性 Git 操作。
