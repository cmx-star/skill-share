# 直接工作流

处理直接模式的工作时，使用此 Skill。写入前遵循全局修改授权规则。

## 入口

- 对话、解释、状态检查、只读盘点、单个命令或微小明显修改，不要创建计划或规划文件。
- 开始需要写入的工作前，说明目标、范围、如何验收和怎么验证。简单改动用一两句话交代；复杂工作再按需要补充不变量、停止条件和其他细节。
- 未经批准，不得重新定义需求、扩大范围、改变公共接口约定、引入依赖或跳过验证。
- 不要因为任务有歧义、规模大、需要调试或审查、风险较高，就自行进入 Flow。先找出真正会影响结果的未知点；只有必要时才问一个聚焦的问题。

## 何时创建计划

工作包含三个及以上有序结果、两个可独立验证的变更目标、昂贵验证阶段、明确计划请求或中断恢复风险时，在首个任务动作前使用 `update_plan`。

- 使用 3-7 个结果导向步骤，且始终只有一个 `in_progress`。
- 仅凭证据标记步骤完成；同时更新当前与下一步骤。
- 不要为单个搜索、读取、工具调用或进度消息创建计划。
- 计划更新后不得停止。继续下一可执行动作，或报告具体阻塞。
- 仅在分别有意义时才拆分实施、定向测试、lint/typecheck、构建和浏览器检查。
- 中断后保留已完成步骤，将命令状态标为 `unknown`，检查证据并只重跑最小缺失检查。

## 高效取证

- 最多分四步取证：建立基线、范围内修改、定向验证和检查 diff。只有新证据会改变决定时，才增加步骤。
- 每次搜索或读取都要有一个明确目的。先找路径和位置，再读取需要的范围；常规输出保持简短。
- 不得根据截断输出推理。立即缩窄查询；同一未变文件不要重复读取超过两次。
- 优先使用 `rg`、结构化 API、专用连接器和项目命令。Shell 只在这些方法不适用时使用。
- 简单任务在 12 次工具调用或两次失败尝试后，整合证据；报告或提出实质问题前至多再做一次有界尝试。
- 复用有效命令结果和 receipt。不得仅为节省 token 使用子代理。

## 实施

1. 从证据定位实现和相邻测试。
2. 按本地模式应用一个最小补丁。
3. 运行定向验证。
4. 检查改动文件或 diff，确认没有超出已批准的范围。
5. 只有结果符合已批准的任务约定时才报告；否则说明失败或不确定性。

修改文档前先收集证据。只改能由代码、配置、测试或明确决定验证的内容。除非批准范围明确要求，否则不要重组、扩写或删除文档；不确定的地方直接说明未知。

## 响应与恢复

- 每次工具返回后报告结果或下一动作。不要发送占位更新。
- 命令状态仅使用 `running`、`passed`、`failed` 或 `unknown`。结果丢失是 `unknown`，不是运行中。
- 长命令后，先报告退出状态和摘要，再做可选检查。仅当新证据指出修正方向时才重试。
- 保留失败尝试。发生冲突或意外变更时，补丁前重新读取。
- 仅在明确请求、责任变化、跨会话工作或未完成停止时交接。包含已确认事实、变更、证据、失败、未知项、风险和一个下一动作。

## 工具选择

- 仓库事实使用本地文件和项目命令；API 使用官方或项目文档。
- 实时/私有状态使用专用连接器。除非用户明确请求内置 Browser，浏览器验证优先 Chrome。
- 每任务只加载一次浏览器说明，批量执行一个连贯场景；仅在需要基线、诊断或最终视觉证据时截图。
- 断言优先于截图。报告简洁的通过/失败结果和失败元素详情。
- MCP/插件变更需要明确范围和批准。不得打印携带凭据的配置。
