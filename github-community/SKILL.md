# GitHub 社区调研

把 GitHub 当作设计取证来源，不要把它当成直接操作的界面。

## 工作流

1. 先检查相关本地代码和项目文档。仅当本地证据无法确定设计时才搜索 GitHub。
2. 搜索前先说清设计问题。查询保持聚焦：框架、行为、接口和一个关键约束。
3. 仅使用 `github` server 暴露的 MCP 工具：`search_repositories`、`search_code` 和 `get_file_contents`。
4. 优先选择官方组织、框架维护者和成熟库；选择有测试、持续维护、许可明确且约束相近的仓库。
5. 只阅读验证模式所需的文件。不要仅将 stars、片段或搜索排名视为质量证明。
6. 将外部模式与当前项目的版本、约定、依赖和兼容性要求对比。
7. 返回仓库、文件路径、相关行为、适用性、取舍和未采用方案等证据。

## 边界

- 不得创建、更新、fork、push、评论、审查、合并或以其他方式改变 GitHub 状态。
- 没有解释清楚冲突前，不要用 GitHub 结果推翻项目本地证据。
- 未检查许可和兼容性前，不得复制大量代码。
- 不请求或打印令牌。若 `GITHUB_PERSONAL_ACCESS_TOKEN` 不可用，报告阻塞。
- 不得静默回退到浏览器搜索或 `gh`；先说明不可用的 MCP 能力。

## 输出

```text
设计问题：
本地证据：
GitHub 参考：
可借鉴的做法：
取舍：
未采用的方案：
许可证或兼容性风险：
```
